# 开发计划

## 阶段 1：前端原型工程化

目标：使用 Vue 3 还原 HTML 原型，接入 mock JSON。

交付：

- Vue 项目脚手架
- 页面路由
- Layout、导航、顶部栏
- 10 个核心页面
- 登录页、用户权限页、更新日志页
- mock 数据接入
- 基础筛选、勾选、抽屉、日志展示
- 路由守卫和 mock 登录态

## 阶段 2：Go 后端 mock API

目标：后端提供稳定 API，前端从 API 获取数据，并提供 mock 登录、权限和更新日志接口。

交付：

- Gin 服务
- REST API 路由
- mock repository
- 统一响应格式
- 基础错误码
- mock auth API
- 用户、角色、权限 mock API
- 更新日志 mock API

## 阶段 3：数据库与任务模型

目标：引入 PostgreSQL 和 Redis，任务状态可持久化。

交付：

- 数据库迁移
- GORM model
- 用户、角色、权限、更新日志、操作日志模型
- 发布单、部署任务 CRUD
- Redis Stream mock Agent worker

## 阶段 4：真实集成预留

目标：为 Jenkins、Harbor、K8s 接入预留 adapter。

交付：

- adapter interface
- mock adapter
- integration config
- 单元测试

## 阶段 5：服务发版与服务部署闭环

目标：区分服务发版和服务部署，形成更贴近真实研发/项目环境的发布逻辑。

外部环境前置要求：

- 进入本阶段前，如需继续接真实链路，需提前准备并告知以下环境是否可用：
  - Jenkins：可访问地址、测试 Job 或可创建测试 Job 的空间
  - Harbor 或兼容镜像仓库：测试项目、测试镜像、读写凭证准备方式
  - Kubernetes 集群：目标 namespace、测试 workload、基础访问方式
- 若上述环境暂未准备，本阶段仍可先以 mock adapter 完成页面与后端流程；一旦研发要切到真实联调，必须先确认环境，不在开发到一半时临时倒推准备。

交付：

- 发布创建页支持选择“服务发版”和“服务部署”
- 服务发版不基于来源基线：
  - 方式 1：Jenkins Job 发版。选择与 Jenkins 视图或特征 job 关联后的 job，执行构建 jar/dist、制作镜像并推送到本地 Harbor
  - 方式 2：本地 Harbor 镜像发版。扫描本地 Harbor 上该服务的镜像版本，选择镜像 tag 发版
- 上述两种服务发版方式最终都需要通过项目环境中运行的 Agent 同步到项目环境，完成镜像同步和 tag 更新
- 服务部署基于基线/生产环境与目标环境对比，识别目标缺失服务
- 目标缺失服务在差异对比页明确标记为“服务部署”
- 服务部署提交后创建部署任务，并通过 Agent task status 展示执行状态和日志
- 发布/部署详情页根据 `agentTaskId` 轮询 Agent 日志；无 `agentTaskId` 时降级展示静态日志
- 预留真实项目环境 Agent 扫描服务列表并上报 RuntimeSnapshot 的后续实现

执行规则：

- 服务发版：目标环境已有服务。Jenkins Job 路径先调用 Jenkins 完成构建、镜像制作和推送本地 Harbor；本地 Harbor 镜像路径直接选择已有镜像 tag；两者最终都通过项目环境 Agent 完成项目 Harbor 镜像同步和 workload tag 更新
- 服务部署：目标环境缺失服务，从来源基线/生产环境确认服务清单后创建部署任务，将服务部署到目标环境
- 混合场景应拆分为 Jenkins 发版任务和服务部署任务，不把两类执行流混在一个任务里
- 后续每当研发即将进入 Jenkins、Harbor、Kubernetes、真实 Agent 联调前，都要先明确向环境提供方提出准备需求，再继续实现或验证真实集成

## 当前研发优先级

当前阶段以 V1 功能闭环为最高优先级，非阻塞性的优化项全部后置。

V1 最低交付目标：

- 平台至少支持远程项目环境的部署与管理
- 平台至少支持远程项目环境的发布任务/部署任务创建、执行跟踪、日志查看和结果反馈

在达到上述目标前，包体优化、构建 warning 清理、纯重构、UI 细节打磨都不应优先于主链路功能开发，除非它们已经阻塞构建、测试或功能交付。

## V1 后续研发主线与环境准备

### Agent 部署方案

- V1 默认方案：Agent 部署在独立 Linux 主机上，使用 `docker compose` 运行。
- V1 默认不把 Agent 自身作为 Kubernetes 内 workload 部署前提。
- V1 中 Agent 的职责是：
  - 访问平台 API 拉取任务、回传状态、回传日志、回传结果
  - 访问 Jenkins 触发或查询构建任务
  - 访问 Harbor 或兼容 Registry 查询、同步、推送镜像
  - 访问目标 Kubernetes 集群执行运行态采集、部署、更新和健康检查
- 后续是否将 Agent 自身迁移到 Kubernetes，放到 V1 主功能稳定后再评估。

### V1 用户视角功能主线

为了便于后续页面联调和测试，V1 主线按“用户能完成什么动作”定义如下：

1. 登录进入平台
   - 用户可以登录、进入工作台、退出登录
   - 用户可以看到自己的身份和角色
2. 管理远程项目环境
   - 用户可以查看环境列表
   - 用户可以查看某个环境的接入方式、Agent 状态、最近任务
3. 采集环境运行态并生成基线
   - 用户可以从来源环境生成基线
   - 用户可以查看基线详情、来源环境、服务数量、锁定状态、用途
4. 对比来源基线与目标环境差异
   - 用户可以看到哪些服务需要发版
   - 用户可以看到哪些服务在目标环境缺失、需要部署
5. 创建发布任务
   - 用户可以对目标环境已有服务发起发布
   - 用户可以选择 Jenkins Job 发版或 Harbor 镜像发版
   - 用户可以提交后进入发布详情追踪执行过程
6. 创建部署任务
   - 用户可以对目标环境缺失服务发起部署
   - 用户可以提交后进入部署详情追踪步骤执行
7. 在详情页追踪执行结果
   - 用户可以查看状态、步骤、日志、失败原因、执行记录、发布报告
   - 用户可以在失败或卡点时执行重试、跳过、人工确认、回滚
8. 查看远程 Agent 执行状态
   - 用户可以看到 Agent 是否在线、最近一次心跳、最近任务结果
9. 查看审计与权限结果
   - 用户可以看到谁对哪个环境执行了什么动作
   - 管理员可以控制环境级操作权限

### 页面验收口径

每个页面至少确认以下 5 件事：

1. 用户能不能进入页面
2. 用户能不能看懂当前状态
3. 用户能不能完成核心动作
4. 动作执行后状态会不会更新
5. 出错时有没有明确提示和兜底展示

### 阶段 A：发布/部署详情闭环

研发目标：

- 补齐发布详情、部署详情、`agentTaskId` 关联
- 展示任务步骤、运行状态、实时日志、失败原因
- 预留重试、跳过、人工确认的状态展示

进入本阶段前建议准备：

- 本地开发环境可正常运行前后端
- 如需提前真实联调：
  - 1 台可运行 `docker` / `docker compose` 的 Linux 主机，作为 Agent 宿主机
  - Agent 宿主机到平台 API 的网络连通性
  - 一个测试环境，用于验证任务状态与日志回传

用户视角交付：

- 在发布详情页看到当前执行状态、步骤、实时日志、执行记录、发布报告
- 在部署详情页看到当前执行状态、步骤、实时日志、执行记录
- 在任务失败或阻塞时，页面上可执行重试、跳过、人工确认、回滚等动作入口

### 阶段 B：Agent 协议闭环

研发目标：

- 实现 Agent 心跳
- 实现 Agent 拉任务
- 实现步骤状态回传
- 实现日志回传
- 实现最终结果回传

进入本阶段前需要准备：

- Agent 宿主机：
  - Linux
  - 已安装 `docker` 与 `docker compose`
  - 具备基础日志查看和配置更新能力
- 网络连通性：
  - Agent 可访问平台 API
  - 平台可识别 Agent 注册和心跳
- 测试对象：
  - 至少一个可反复执行的测试项目或测试服务

用户视角交付：

- 在 Agent 管理页看到 Agent 注册、在线状态、最近心跳
- 发布/部署任务可以由 Agent 实际拉取并执行
- 详情页中的步骤和日志不再只是 mock 轮询，而是来自实际 Agent 回传

下一步需要做什么：

- 补齐 Agent 注册、心跳、拉任务、步骤状态回传、日志回传、结果回传接口与页面展示
- 打通平台任务与 Agent 执行之间的最小闭环
- 若要做真实验证，需要你先准备本阶段所列 Agent 宿主机与网络条件

### 阶段 C：真实发布链路联调

研发目标：

- 打通 Jenkins Job 发版链路
- 打通 Harbor 镜像选择与同步链路
- 打通 Agent 执行镜像同步与目标环境 tag 更新链路

进入本阶段前必须准备：

- Jenkins：
  - 可访问实例
  - 测试 Job，或可创建测试 Job 的空间
  - 凭证接入方式已明确
  - Job 命名规则或视图规则已明确
  - 至少 1 条可重复执行的测试流水线
  - 对应的 Jenkinsfile 或构建脚本已准备
  - 该流水线可以完成代码拉取、构建产物、构建镜像、推送 Harbor
- Harbor 或兼容 Registry：
  - 测试项目
  - 可读写权限
  - 一个或多个测试镜像
  - 测试 tag 使用规则
  - 至少 1 个可用于 K8s 部署的测试镜像
  - 最好准备 2 个以上 tag，用于验证版本切换和更新
- Agent 宿主机：
  - 可访问 Jenkins
  - 可访问 Harbor / Registry
  - 可访问平台 API

建议同时准备的最小联调样本：

- 1 个测试服务源码仓库
- 该服务的 `Dockerfile`
- 该服务的 Jenkinsfile 或构建脚本
- 已推送到 Harbor 的测试镜像与 tag

用户视角交付：

- 用户可以在创建发布页选择 Jenkins Job 发版或 Harbor 镜像发版
- 用户可以看到镜像来源、目标环境、执行进度和最终结果
- 用户不需要手工去项目 Harbor 查 tag，也不需要手工改 workload tag

### 阶段 D：真实部署链路联调

研发目标：

- 打通运行态采集
- 打通目标缺失服务部署
- 打通 workload 更新、健康检查、结果回传

进入本阶段前必须准备：

- Kubernetes：
  - 可访问测试集群
  - 测试 namespace
  - 至少一个可测试 workload
  - `kubeconfig` 或服务账号准备方式已明确
  - 至少 1 套可重复部署的 K8s manifests
  - 最好同时具备“已有服务更新 tag”和“缺失服务首次部署”两类验证样例
- Agent 宿主机：
  - 可访问 Kubernetes API
  - 可执行 kubernetes 相关部署与查询动作
- 联调用例：
  - 至少一个“目标环境缺失服务”的可重复验证场景

建议同时准备的最小联调样本：

- 1 个测试服务对应的 Deployment/StatefulSet manifests
- 如有需要，对应的 Service、ConfigMap、Ingress 等最小配套资源
- 1 套用于验证已有服务更新镜像 tag 的 manifests
- 1 套用于验证目标环境缺失服务首次部署的 manifests

用户视角交付：

- 用户可以从差异页直接识别“目标环境缺失服务”
- 用户可以发起部署并由 Agent 在目标环境完成部署与健康检查
- 用户可以在部署详情页看到每一步结果和失败原因

### 阶段 E：远程项目环境部署与管理达标

研发目标：

- 远程环境管理可用
- Agent 注册、状态、最近任务可见
- 远程发布与远程部署都可从平台发起并追踪
- 达到 V1 最低可交付标准

进入本阶段前必须准备：

- 一整套联调环境：
  - Agent `docker-compose` 宿主机
  - Jenkins
  - Harbor / Registry
  - Kubernetes
  - 测试项目或测试服务
  - 测试服务对应的镜像、Jenkins 流水线、构建脚本、K8s manifests
- 基础协作信息：
  - 环境准备负责人
  - 权限申请路径
  - 测试窗口或联调时间安排

用户视角交付：

- 用户可以在平台里管理远程项目环境
- 用户可以在平台里发起远程发布和远程部署
- 用户可以在平台里追踪 Agent、任务、结果、日志，达到 V1 可交付标准

## 阶段 7：差异分类与远程任务闭环

目标：把差异结果正式映射为发布或部署动作，并通过 Agent 任务流闭环执行。

交付：

- 后端统一判定差异分类

- `MISSING_IN_TARGET` 映射到部署任务
- 发布/部署创建结果保留 `agentTaskId`
- 详情页展示任务状态、步骤、日志、失败原因
- 重试、跳过、人工确认等状态流转可见

## 阶段 8：远程项目环境部署与管理达标

目标：达到 V1 最低可交付标准，即可管理远程项目环境并完成远程部署/发布主流程。

外部环境前置要求：

- 进入本阶段前，需要至少有一套可用于联调的远程项目环境，且提前确认：
  - Agent 部署位置与运行方式
  - Jenkins、Harbor、Kubernetes 的连通性
  - 一个可反复执行的测试项目/测试服务
  - 基础账号、权限申请路径和环境准备负责人
- 如果这些条件未满足，应先停留在 mock / 本地验证，不把“真实远程部署与管理达标”标记为完成。

交付：

- 项目环境管理和连接状态可见
- Agent 注册、心跳、能力、最近任务可见
- Agent 拉任务、上报步骤状态、上报日志、上报最终结果
- 远程部署任务和远程发布任务均可从平台发起并追踪
- 操作审计记录完整覆盖操作人、环境、服务、tag 变化、结果和失败步骤
