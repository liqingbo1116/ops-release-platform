# 开发计划

## 历史原型阶段（已废弃）

本节只保留早期原型背景，不作为当前 V1 执行顺序。当前 V1 禁止使用模拟数据、模拟仓储、模拟 adapter、模拟 Agent worker 或前端模拟兜底作为研发交付标准。

### 阶段 1：前端原型工程化

交付：

- Vue 项目脚手架
- 页面路由
- Layout、导航、顶部栏
- 10 个核心页面
- 登录页、用户权限页、更新日志页
- 真实后端 API 接入
- 基础筛选、勾选、抽屉、日志展示
- 路由守卫和真实后端登录态

### 阶段 2：Go 后端 API

目标：后端提供稳定 API，前端从 API 获取数据，并提供真实登录、权限和更新日志接口。

交付：

- Gin 服务
- REST API 路由
- PostgreSQL repository
- 统一响应格式
- 基础错误码
- auth API
- 用户、角色、权限 API
- 更新日志 API

### 阶段 3：数据库与任务模型

目标：引入 PostgreSQL 和 Redis，任务状态可持久化。

交付：

- 数据库迁移
- GORM model
- 用户、角色、权限、更新日志、操作日志模型
- 发布单、部署任务 CRUD
- 真实 Agent 任务租约和状态回传模型

### 阶段 4：真实集成

目标：接入 Jenkins、Harbor、K8s 真实 adapter。

交付：

- adapter interface
- real adapter
- integration config
- 单元测试

## 阶段 5：服务发版与服务部署闭环

目标：区分服务发版和服务部署，形成更贴近真实研发/项目环境的发布逻辑。

外部环境前置要求：

- 进入本阶段前，如需继续接真实链路，需提前准备并告知以下环境是否可用：
  - Jenkins：可访问地址、测试 Job 或可创建测试 Job 的空间
  - Harbor 或兼容镜像仓库：测试项目、测试镜像、读写凭证准备方式
  - Kubernetes 集群：目标 namespace、测试 workload、基础访问方式
- 若上述环境暂未准备，本阶段必须标记阻塞并提示需要准备的资源，不能以模拟 adapter 完成页面与后端流程。

交付：

- 发布创建页支持选择“服务发版”和“服务部署”
- 服务发版不基于来源基线：
  - 方式 1：Jenkins Job 发版。选择与 Jenkins 视图或特征 job 关联后的 job，执行构建 jar/dist、制作镜像并推送到本地 Harbor
  - 方式 2：本地 Harbor 镜像发版。扫描本地 Harbor 上该服务的镜像版本，选择镜像 tag 发版
- 上述两种服务发版方式最终都需要通过项目环境中运行的 Agent 同步到项目环境，完成镜像同步和 tag 更新
- 服务部署基于基线/生产环境与目标环境对比，识别目标缺失服务
- 目标缺失服务在差异对比页明确标记为“服务部署”
- 服务部署提交后创建部署任务，并通过 Agent task status 展示执行状态和日志
- 发布/部署详情页根据 `agentTaskId` 轮询 Agent 日志；无 `agentTaskId` 时降级展示静态日志
- 预留真实项目环境 Agent 扫描服务列表并上报 RuntimeSnapshot 的后续实现

执行规则：

- 服务发版：目标环境已有服务。Jenkins Job 路径先调用 Jenkins 完成构建、镜像制作和推送本地 Harbor；本地 Harbor 镜像路径直接选择已有镜像 tag；两者最终都通过项目环境 Agent 完成项目 Harbor 镜像同步和 workload tag 更新
- 服务部署：目标环境缺失服务，从来源基线/生产环境确认服务清单后创建部署任务，将服务部署到目标环境
- 混合场景应拆分为 Jenkins 发版任务和服务部署任务，不把两类执行流混在一个任务里
- 后续每当研发即将进入 Jenkins、Harbor、Kubernetes、真实 Agent 联调前，都要先明确向环境提供方提出准备需求，再继续实现或验证真实集成

## 当前研发优先级

当前阶段以 V1 功能闭环为最高优先级，非阻塞性的优化项全部后置。

V1 最低交付目标：

- 平台至少支持项目环境的部署与管理
- 平台至少支持项目环境的发布任务/部署任务创建、执行跟踪、日志查看和结果反馈

在达到上述目标前，包体优化、构建 warning 清理、纯重构、UI 细节打磨都不应优先于主链路功能开发，除非它们已经阻塞构建、测试或功能交付。

## V1 研发主线路径

本节替代旧的 mock-first 研发顺序，作为当前唯一有效的 V1 研发路径。要求如下：

- 只做主线，不插入优化项、重构项、体验打磨项。
- 每个阶段必须切到真实数据，不能以“先保留 mock，后面再替换”作为阶段完成标准。
- 当前阶段的 mock 数据、mock repository、mock fallback 没删掉，就不能进入下一阶段。
- 如果替换 mock 依赖外部工具或运行环境，这些前置条件没有准备好，则本阶段不能算完成，下一阶段也不能开始。
- 本地开发启动方式仍然是 frontend `npm run dev`、backend `go run`，但数据依赖必须接真实 PostgreSQL、真实 Redis，以及各阶段要求的真实外部系统。

### 阶段 1：基础资源管理

目标：

- K8s 集群、Harbor 仓库、Jenkins 实例作为平台独立资源维护，不能只藏在 `.secrets/` 或环境记录里。
- 新增资源时按用户能理解的字段填写，不让用户填写 `credentialRef`：
  - K8s：名称、kubeconfig 上传或粘贴、可选 context。
  - Harbor：名称、地址、HTTP/HTTPS、用户名、密码、可选跳过 TLS 校验。
  - Jenkins：名称、地址、用户名、密码或 API Token、可选跳过 TLS 校验。
- 凭据由后端保存到正式凭证后端或研发阶段私密来源，资源表只保存内部 `credentialRef`。
- 资源状态由系统维护，用户不能手工改状态。
- 每类资源保留“测试连接”和“刷新/重新探测”能力。
- 探测结果缓存到平台：K8s namespaces、Harbor projects、Jenkins views/jobs。
- 刷新失败时保留旧缓存，并展示失败原因。
- 删除资源管理页面和接口中的 mock 数据与 fallback。

状态：

- 进行中。
- 已完成平台基础资源拆分、真实数据库持久化、用户视角表单、系统维护状态、测试连接、刷新探测和缓存字段。
- 已完成本地/直连资源探测：K8s `/readyz` 与 namespace 列表、Harbor systeminfo 与 project 列表、Jenkins 基础 API 与 view/job 列表。
- 刷新失败时保留旧缓存，并在资源上更新失败原因。
- 项目环境资源通过 Agent 代理探测并回传状态/缓存仍需补齐；V1 Agent 只探测项目环境 K8s/Harbor，不连接 Jenkins。
- 不能再把“平台直连远程 K8s/Harbor 检查通过”作为远程资源验收标准；远程项目环境应由 Agent 执行探测并回传。

必须准备：

- 前端本地运行环境：Node.js、npm
- 后端本地运行环境：Go
- 远程 PostgreSQL
- 远程 Redis
- `.secrets/` 中可用的研发阶段私密连接配置
- 可用于本地直连验证的 K8s kubeconfig
- 可用于本地直连验证的 Harbor 地址与账号密码，必须支持 HTTP Harbor
- 可用于本地直连验证的 Jenkins 地址与账号密码或 API Token
- 项目环境如需探测，必须先具备可上线的 Agent、Agent 到平台 API 的出站网络，以及 Agent 侧访问项目 K8s/Harbor 的网络和凭据来源

门禁：

- 如果 PostgreSQL 或 Redis 未准备好，基础资源管理无法替换 mock，不能进入阶段 2。
- 如果资源新增仍要求用户填写 `credentialRef`，本阶段未完成。
- 如果资源状态仍由用户手工维护，本阶段未完成。
- 如果 K8s namespace、Harbor project、Jenkins view/job 没有真实探测与缓存，本阶段未完成。
- 如果缓存没有“刷新/重新探测”入口，本阶段未完成。
- 如果远程资源仍由平台后端直接连远程网络探测，而不是通过 Agent 任务探测，本阶段未完成。

### 阶段 2：环境管理

目标：

- 环境列表、环境详情、环境创建、环境编辑、环境状态全部改为真实后端数据。
- 页面只让用户选择“本地环境”或“远程环境”，不暴露 `DIRECT` / `AGENT` 网络模式选项。
- 环境增加部署目标类型 `deployTargetType`：V1 当前实现 `KUBERNETES`，模型预留 `DOCKER_COMPOSE`，但 docker-compose 执行不进入当前阶段。
- 本地环境内部固定为 `DIRECT`，由平台直连基础资源。
- 远程环境内部固定为 `AGENT`；项目 K8s/Harbor 由 Agent 探测和执行，平台不直连项目 K8s。
- 环境不维护 K8s/Harbor/Jenkins 凭据，只关联阶段 1 的资源。
- 环境通过资源绑定模型关联阶段 1 的资源，一个环境可关联多个作用域；当前页面先维护每类资源的默认绑定，后续多选编辑在此模型上扩展。
- V1 环境管理阶段必须先完成多作用域绑定的模型、后端接口和数据库持久化，不能继续把环境设计成只能绑定一个 K8s namespace、一个 Harbor project、一个 Jenkins view/job。
- 多作用域绑定的用户价值是服务关联：后续用户在环境内维护服务时，可以把服务关联到实际使用的 namespace、Harbor project；本地环境还可以关联 Jenkins view/job 作为构建来源。发布/部署时平台据此选择构建来源、镜像范围和部署目标。不能把多绑定做成只有环境配置页可见、后续业务不用的孤立配置。
- 本地环境通过资源 ID 关联 K8s/Harbor/Jenkins，并填写环境级作用域：
  - K8s `namespace`
  - Harbor `registryProject`
  - Jenkins `jenkinsView`
- 项目环境关联项目 K8s namespace 和项目 Harbor project，由 Agent 后续上报、验证和执行；项目环境不绑定 Jenkins view，Jenkins 属于平台侧本地基础资源，用于后续服务/版本来源阶段的本地构建。
- 同一个 K8s、Harbor、Jenkins 可以被多个本地环境复用，不同环境可使用不同 namespace/project/view；项目环境可以绑定多个 namespace/project。Jenkins 只在本地环境和服务构建来源中使用。
- 页面交互可先只维护每类资源的默认绑定，但环境详情接口必须返回完整 bindings 列表；发布/部署创建时必须记录实际选用的 namespace/project/view，避免后续环境绑定变化影响历史任务。
- 创建环境时面向用户填写环境名称，系统自动生成可见的“环境标识”；用户可按需修改标识，环境 ID 由系统按 `env-环境标识` 生成，不要求用户同时维护两套标识。
- 选择 namespace/project/view 时优先使用阶段 1 的缓存列表，同时允许手工输入作为兜底；如果手工输入的作用域不在最近探测缓存中，环境允许保存但状态必须进入 `DEGRADED`，前端显示“需验证”，直到资源探测补齐或连接检查确认。
- 删除环境管理页面和后端接口中的环境 mock 数据与 fallback。

必须准备：

- 阶段 1 已完成
- 至少一个真实 K8s 资源及 namespace 缓存，用于创建本地环境
- 至少一个真实 Harbor 资源及 project 缓存，用于创建本地环境；项目环境 Harbor project 由 Agent 探测验证
- Jenkins 如进入本地环境或发布来源验收，则必须有真实 Jenkins 资源及 view/job 缓存；项目环境不需要 Jenkins 绑定

门禁：

- 如果环境仍内嵌维护 K8s/Harbor/Jenkins 凭据，本阶段未完成。
- 如果本地环境关联的 namespace/project/view 仍来自 mock，本阶段未完成。
- 如果环境资源绑定仍是单 namespace、单 Harbor project、单 Jenkins view/job 的单值模型，本阶段未完成。
- 如果环境多绑定没有为后续服务关联预留模型或接口入口，本阶段未完成。
- 如果发布/部署任务不能保存实际选用的 namespace/project/view 快照，本阶段未完成。
- 如果环境创建仍要求用户同时理解和填写环境 ID 与环境标识，或环境标识不能自动生成，本阶段未完成。
- 如果远程环境仍允许平台后端直连远程 K8s 或执行远程直连检查，本阶段未完成。

### 阶段 3：Agent 管理与远程探测

目标：

- Agent 注册、心跳、环境绑定、在线状态、最近任务状态全部改为真实数据。
- Agent 管理页面不再依赖 mock Agent 列表。
- Agent 管理页面可以生成一次性注册密钥，并展示平台地址、密钥有效期和二进制启动配置示例。
- 项目现场人员把平台地址和注册密钥写入 Agent 配置文件后，用二进制方式启动 Agent。
- Agent 首次注册成功后由平台签发长期 Agent token，一次性注册密钥不得用于运行期接口。
- Agent 支持先凭平台地址和注册密钥接入平台，平台能展示未绑定/待认领 Agent，再由平台绑定到项目/产品；当前环境记录在 V1 中承接产品部署范围。
- Agent 启动时不绑定项目环境；页面认领后由平台绑定到项目/产品，运行中的 Agent 通过心跳同步绑定关系。
- Agent 心跳、任务租约、远程探测和执行回传接口必须校验长期 Agent token，不能只生成 token 不校验。
- Agent 处于 `在线 / 待认领` 时可以展示未归属探测摘要，但不能进入正式产品/服务视图，也不能执行发布/部署任务。
- 项目/远程环境的资源测试连接和刷新探测通过 Agent 任务执行。
- Agent 只探测项目环境 K8s namespace 和 Harbor project 后回传状态和缓存；Jenkins 由平台后端直连本地基础资源，Agent 不连接 Jenkins、不触发构建、不制作镜像。

必须准备：

- 阶段 1、阶段 2 已完成
- 平台中已经可以真实创建环境记录
- 可构建的 agent 二进制
- agent 支持通过 `-f` 指定配置文件直接启动
- 一台可出站访问平台 API 的远程 Linux 主机
- 平台可访问的 `PLATFORM_URL`
- Agent 侧具备访问项目 K8s/Harbor 的网络和凭据来源；本地 Jenkins、本地 Harbor 已在平台基础资源中维护

门禁：

- 如果 Agent 只能在启动时绑定环境，平台不支持未绑定/待认领 Agent，本阶段未完成。
- 如果注册密钥不能从页面生成，或不能一次性注册后换长期 Agent token，本阶段未完成。
- 如果 Agent token 没有在心跳、任务租约、远程探测和执行回传接口校验，本阶段未完成。
- 如果待认领 Agent 的探测数据进入正式产品/服务视图，或待认领 Agent 可执行发布/部署任务，本阶段未完成。
- 如果 Agent 管理仍依赖 mock Agent、mock 探测数据或 mock 执行结果，本阶段未完成。
- 如果远程 Linux 主机、二进制启动方式、或平台出站连通性未准备好，Agent 管理无法替换 mock，不能进入阶段 4。
- 如果远程资源探测不能通过 Agent 回传状态和缓存，不能进入阶段 4。

### 阶段 4：项目管理

目标：

- 补齐用户视角的最高层级：项目 -> 产品 -> 服务 -> 发布 / 部署。
- 项目是业务归属边界，例如项目A、项目B。
- 项目作为产品、服务、发布单、部署单、基线和权限的上层归属入口。
- 项目可以绑定一个或多个产品；产品在 V1 中最多归属一个项目。
- 已完成的基础资源管理和环境管理不推翻重做，本阶段先补齐项目实体、项目列表、项目详情、项目选择和项目-产品绑定入口。
- 基础资源管理继续作为全局资源池；K8s、Harbor、Jenkins 不直接归属项目，也不由项目直接绑定。
- 产品引用基础资源范围；项目通过已绑定产品间接看到资源使用情况。
- 界面上至少要把环境管理的主业务展示调整为产品管理：用户看到的是产品，例如数据中台、物联中台；环境已有的资源绑定、状态、Agent 就绪能力作为产品部署范围继续复用。
- 项目详情需要能看到已绑定产品、可绑定产品、产品绑定状态，以及绑定/解绑/更换项目入口。
- 产品需要支持几种绑定状态，用于项目绑定、解绑或迁移：
  - `UNBOUND`：未绑定项目，可被项目认领。
  - `BOUND`：已绑定项目，可进入服务、发布、部署流程。
  - `MOVING`：正在更换项目，临时禁止创建新的发布/部署任务。
  - `DISABLED`：停用，不进入新的发布/部署选择。
- 解绑或更换项目不能影响历史发布/部署/基线记录；历史记录必须保留当时的项目、产品、服务快照。

必须准备：

- 阶段 1 到阶段 3 已完成
- 环境数据已经真实持久化
- Agent 可以绑定到项目/产品或展示为未绑定/待认领
- 已有本地环境和项目环境数据可作为产品数据迁移或映射验证样本

门禁：

- 如果没有真实项目实体和持久化，不能进入阶段 5。
- 如果页面仍只展示“环境管理”而没有面向用户的“产品管理”入口，不能进入阶段 5。
- 如果项目不能绑定多个产品，不能进入阶段 5。
- 如果产品没有绑定状态，不能支持未绑定、已绑定、迁移中、停用等用户可理解状态，不能进入阶段 5。
- 如果项目只是页面筛选项，没有被后续产品、服务、发布、部署、基线、权限模型消费，不能算完成。

### 阶段 5：产品管理

目标：

- 补齐项目下的产品层级，例如数据中台、物联中台。
- 当前“环境”概念在 V1 中对应产品概念：环境已有的资源绑定、状态、Agent 就绪等能力作为产品部署范围的过渡实现。
- 不在产品下面再额外暴露一层环境；后续服务直接归属产品，并消费产品范围内的 K8s namespace、Harbor project、Jenkins view/job。
- 已完成的环境管理不推翻重做，本阶段把环境数据承接为产品数据或产品部署范围数据。
- 产品列表/详情展示产品名称、所属项目、绑定状态、资源范围、Agent 状态、服务数量和最近更新时间。
- 产品详情只展示用户判断产品是否可用所需的信息：资源范围是否准备好、Agent 是否准备好、服务数量和最近上报摘要；资源绑定明细可以折叠到详情中。
- 产品绑定到项目后，后续服务、发布、部署、基线都必须从项目 -> 产品进入，不能再直接从环境进入。

必须准备：

- 阶段 1 到阶段 4 已完成
- 项目实体已真实持久化
- 环境数据已经真实持久化，并可承接为产品部署范围
- 已定义产品绑定状态和项目-产品绑定/解绑/迁移规则
- 基础资源继续作为全局资源维护，产品只引用资源范围，不把资源复制或迁移到项目下

门禁：

- 如果产品没有项目归属，不能进入阶段 6。
- 如果当前环境能力没有被明确承接为产品部署范围，导致用户视角仍是“产品下面还有环境”，不能进入阶段 6。
- 如果产品不能展示和维护绑定状态，不能进入阶段 6。
- 如果 K8s、Harbor、Jenkins 被设计成项目直绑资源，而不是产品引用的全局资源，不能进入阶段 6。

### 阶段 6：服务与版本来源

目标：

- 产品服务列表、Jenkins Job、Harbor 镜像 tag、运行态服务版本来源全部改为真实数据。
- 产品服务列表是服务发版的主入口，用户应先按项目/产品定位服务，再从服务行上直接发起发版。
- 服务必须归属到产品，并在产品部署范围内选择实际使用的 K8s namespace、Harbor project、Jenkins view/job。
- 服务纳管页面定位为服务维护页，不作为日常发版主入口；用户可从产品服务列表进入纳管/移除纳管。
- 服务发版不允许手工录入服务或镜像来源，必须基于平台直连或 Agent 上报的真实 workload、镜像和 Jenkins Pipeline 列表选择。
- 服务部署的目标缺失服务来自真实运行态或真实基线，不来自 mock。
- 删除服务、版本、Job、镜像 tag 的 mock 数据与 fallback。

服务列表设计要求：

- 按 `项目 -> 产品 -> 服务` 展示服务。
- 支持按服务名、namespace、workload 类型、镜像来源、Pipeline 绑定状态、可发版状态过滤。
- 展示服务名、Deployment/StatefulSet/DaemonSet 类型、namespace、当前镜像、当前 tag、镜像来源、私有 registry 确认状态、Jenkins Pipeline 绑定状态、最近一次发版状态。
- 每个服务提供发版、绑定/更换 Pipeline、查看详情、移除纳管入口。
- 纳管页展示探测到的 workload，支持多选纳管和多选移除；移除纳管只解除平台管理关系，不删除 K8s workload、Harbor 镜像、Jenkins Pipeline 或历史发版记录。
- 服务列表以真实产品环境为准，不把平台数据库中的纳管关系当作服务主数据：
  - 本地产品以平台直连 K8s/Harbor/Jenkins 探测结果为准。
  - 远程产品以 Agent 上报的 K8s/Harbor 运行态和平台直连本地 Jenkins/Harbor 数据为准。
  - 已纳管服务如果在真实产品环境中被删除或消失，平台必须在收到最新探测数据时自动解除纳管关系，主服务列表不再展示该服务，也不得允许继续发版。
  - 远程产品由 Agent 主动上报运行态快照；平台收到 Agent 上报后必须主动处理纳管关系、资源变化和服务展示状态，不能依赖用户手工点击刷新后才修正。
  - 本地产品可以由页面刷新或后台探测触发平台直连探测，但一旦平台拿到最新探测结果，也必须立即按结果处理纳管关系和服务展示状态。
  - 未纳管服务如果在真实产品环境中被删除或消失，平台在收到最新探测数据后不再展示。
  - 平台可以通过事件日志/资源变化记录展示“已纳管服务消失、已自动解除纳管、服务重新出现”等变化，但不能把已消失服务作为可操作服务继续保留。
  - V1 平台不提供真实删除能力，不删除 K8s workload、Harbor 镜像、Jenkins Pipeline 或 Git 配置；页面文案必须使用“移除纳管/解除纳管”，避免让用户理解为删除真实服务。

Jenkins Pipeline 绑定要求：

- Pipeline 列表来自产品绑定的 Jenkins view/job 缓存或真实 Jenkins 查询。
- 服务第一次发版前如未绑定 Pipeline，必须提示用户先从列表选择，不允许手工录入 Pipeline 名称。
- 平台可以按服务名、workload 名、镜像名推荐 Pipeline，但最终由用户确认。
- 绑定关系保存到服务级配置，后续发版默认复用，并支持在服务详情中更换。

必须准备：

- 阶段 1 到阶段 5 已完成
- 环境与 Agent 已真实关联
- 项目和产品归属已真实持久化
- 真实服务版本来源已经可用，至少一种：
  - Jenkins Job 元数据
  - Harbor/Registry 镜像 tag
  - Kubernetes 运行态镜像版本
  - 平台已持久化的真实运行态快照

门禁：

- 如果服务、Job、镜像 tag 或运行态版本仍是 mock，不能进入阶段 7。
- 如果服务没有产品归属，或没有消费产品部署范围中的 namespace/project/view/job，不能进入阶段 7。
- 如果用户不能从产品服务列表直接定位服务并进入发版，不能进入阶段 7。
- 如果服务发版仍要求用户手工录入服务、镜像或 Pipeline，不能进入阶段 7。
- 如果真实产品环境中已删除/消失的服务仍在平台主服务列表中显示为可发版，或仍保留为有效纳管关系，不能进入阶段 7。
- 如果远程产品必须由用户点击刷新后才处理 Agent 已上报的最新服务变化，不能进入阶段 7。

### 阶段 7：发布单创建

目标：

- 发布入口从产品服务列表进入，用户点击服务上的发版按钮后自动创建发布单。
- 发布单是流程记录和执行详情载体，不是用户寻找服务的主入口。
- 发布单创建必须从真实项目、真实产品、真实 Agent、真实服务、真实 Jenkins Pipeline、真实 Harbor 镜像来源读取数据。
- 发布单创建时生成固定 V1 流程节点，并为后续审批、人工确认、灰度等节点预留扩展模型。
- 删除发布单表单中的 mock 服务、mock 版本、mock 环境依赖。

V1 发版流程：

- Jenkins 执行前平台可以控制流程，例如后续插入审批或人工确认。
- Jenkins 执行中平台只观察状态和日志，不控制暂停，也不插入中间节点。
- Jenkins 结束后，无论本地产品还是远程产品，平台都必须查询本地 Harbor，确认目标镜像 tag 是否已经推送到本地 Harbor project。
- Jenkins 失败时也允许执行本地 Harbor 确认；如果镜像不存在，发版失败或停止后续流程；如果镜像已存在，页面应明确提示“流水线失败但镜像已存在”，默认不自动继续。

本地产品流程：

1. 用户从产品服务列表点击服务发版。
2. 平台创建发布单并触发已绑定 Jenkins Pipeline。
3. Jenkins 完成编译构建、制作镜像、推送本地 Harbor、修改 GitLab YAML。
4. 平台获取 Jenkins build number、执行状态和 console log。
5. 平台确认本地 Harbor project 中存在目标镜像 tag。
6. 平台连接本地 K8s，确认目标 workload 当前镜像 tag 或 rollout 结果。
7. 平台展示 Jenkins、Harbor、K8s 三类结果，最终判定发版成功或失败。

远程产品流程：

1. 用户从产品服务列表点击服务发版。
2. 平台创建发布单并触发已绑定 Jenkins Pipeline。
3. Jenkins 完成编译构建、制作镜像、推送本地 Harbor。
4. 平台获取 Jenkins build number、执行状态和 console log。
5. 平台确认本地 Harbor project 中存在目标镜像 tag。
6. 只有本地 Harbor 镜像确认存在后，平台才允许下发 Agent 远程发版任务。
7. Agent 通过远程 Harbor 的 pull/复制能力从本地 Harbor 拉取镜像，并确认远程 Harbor 中存在目标镜像 tag。
8. Agent 连接远程 K8s 更新目标 workload 镜像，并回传状态、日志和最终结果。
9. 平台展示 Jenkins、本地 Harbor、远程 Harbor、远程 K8s 和 Agent 执行结果。

流程节点模型要求：

- 本地产品固定节点：创建发布单、触发 Jenkins、获取 Jenkins 日志、确认本地 Harbor 镜像、确认本地 K8s 运行结果、完成。
- 远程产品固定节点：创建发布单、触发 Jenkins、获取 Jenkins 日志、确认本地 Harbor 镜像、下发 Agent 任务、远程 Harbor 拉取/确认镜像、远程 K8s 更新、完成。
- 远程产品如本地 Harbor 镜像不存在，必须停止在 Agent 节点之前，不能继续远程发版。
- 发布详情页必须展示每个节点的状态、时间、失败原因和关键日志摘要。

必须准备：

- 阶段 1 到阶段 6 已完成
- 环境与 Agent 已真实关联
- 真实服务发版来源或真实服务部署来源可用
- 平台后端可访问本地 Jenkins 和本地 Harbor
- 本地产品如需验收发版结果，平台后端可访问本地 K8s
- 远程产品如需验收远程发版，Agent 已绑定产品，并可访问远程 Harbor 与远程 K8s

门禁：

- 如果发布单创建仍依赖 mock 项目、mock 产品、mock Agent、mock 服务或 mock 版本，不能进入阶段 8。
- 如果用户仍必须先进入发布单页面再找服务，不能进入阶段 8。
- 如果平台不能获取 Jenkins 执行状态和日志，不能进入真实发版验收。
- 如果 Jenkins 后没有本地 Harbor 镜像确认节点，不能进入真实发版验收。
- 如果远程产品在本地 Harbor 镜像不存在时仍会下发 Agent 任务，不能进入阶段 8。

门禁：

门禁：

### 阶段 10：发布详情 / 部署详情

目标：

- 详情页展示真实项目、产品、服务、任务状态、步骤、日志、失败原因、最终结果。
- 删除详情页对 mock task 记录和 mock 日志的依赖。

## 历史 Agent-first 规划（已被 V1 研发主线路径替代）

本节保留为历史上下文，不再作为后续研发执行顺序。当前唯一有效顺序以上文“V1 研发主线路径”为准：基础资源管理、环境管理、Agent 管理与远程探测、项目管理、产品管理、服务与版本来源、发布单创建、基线管理、部署执行、发布/部署详情、登录与权限、清理剩余 mock。每个阶段必须接真实数据并删除该阶段 mock/fallback，未满足阶段门禁时不能进入下一阶段。

### V1 边界与网络前提

- 环境分为本地环境与项目环境，两者的接入方式必须明确区分。
- 本地环境默认平台侧可以连通，走平台后端 adapter 直连验证，不需要 Agent，也不能把 Agent 作为本地环境前置条件。
- 项目环境默认平台侧不可连通，必须通过部署在项目环境内或可访问项目环境的 Agent 接入。
- 平台不能依赖访问项目环境 Agent endpoint，也不能向 Agent 主动推送任务。
- Agent 只支持出站访问平台 API：主动领取/租约获取任务 payload，并主动上报心跳、服务列表、镜像版本、运行态快照、任务状态、日志和最终结果。
- V1 研发阶段 Agent 部署方式固定为项目环境侧 Linux 主机 + 二进制启动，不要求先把 Agent 自身部署进 Kubernetes。
- `docker compose` 只作为后续正式上线部署方式，不作为当前研发阶段门禁。
- Agent 当前阶段不能用 mock 数据作为完成标准；缺真实 Agent、网络、配置或外部资源时，要记录阻塞并及时说明需要用户准备什么。

### 当前状态

- 已有可运行的独立 Agent 进程：`agent/cmd/agent`。
- 已有 Agent 配置读取、健康检查、心跳上报、任务租约领取客户端、步骤/日志/结果回传客户端。
- Agent 执行器必须连接真实 Jenkins、Harbor 或 Kubernetes；缺少外部资源时只回传不可用状态和具体错误。
- 已有 Agent 专用 `Dockerfile`、`docker-compose.yml` 和 `.env.example`。
- 已有平台侧 `/api/agent-tasks/lease` 出站任务租约接口，创建发布/部署任务时会绑定 `agentId` 与 `environmentId`。
- 已有任务租约超时回收、同一 Agent 单任务租约约束和重复领取防并发策略。
- 已有 `agent/README.md` 说明远程 Agent 二进制部署、配置和真实探测流程。
- 本地验证已通过：
  - `agent`: `go test ./...`
  - `backend`: `go test ./...`
- 真实项目环境发版/部署测试仍需先在远程 Linux 主机上部署 Agent，并验证 Agent 到平台 API 的出站连通性。

### V1 后续执行顺序

1. 远程 Agent 独立部署包（本地实现已完成，待远程主机验证）
   - 交付：可运行 Agent 进程、配置读取、健康检查、任务领取客户端、结果回传客户端、二进制启动配置示例、`.env.example`。
   - 用户视角：环境负责人可以在远程 Linux 主机执行 Agent 二进制启动，平台可以看到 Agent 在线或健康。
   - 环境准备：Linux 主机、Agent 到平台 API 的出站连通性。
   - 不需要：Jenkins、Harbor/Registry、Kubernetes。
2. Agent 主动领取/租约任务链路（本地实现已完成，待远程主机验证）
   - 交付：环境与 Agent 绑定、待执行任务 payload、领取/租约接口、领取确认、租约超时、幂等执行键、状态/日志/结果回传。
   - 用户视角：提交发布/部署后，详情页能看到任务被远程 Agent 领取、执行和回传结果。
   - 环境准备：第 1 步 Agent 可远程运行，Agent 可访问平台 API，可重复提交的真实测试任务。
   - 不需要：Jenkins、Harbor/Registry、Kubernetes。
3. Agent 管理与远程探测真实闭环
   - 交付：页面生成一次性注册密钥、Agent 首次注册、长期 Agent token 签发和校验、心跳、`在线 / 待认领`、项目/产品绑定、远程资源探测和缓存回传。
   - 用户视角：用户在 Agent 页面生成密钥，项目侧用二进制启动 Agent；平台看到 `在线 / 待认领`，可查看未归属探测摘要，绑定到项目/产品后才进入正式任务。
   - 环境准备：项目环境 Agent 主机、平台 API 出站网络、Agent 访问项目 K8s namespace 和项目 Harbor project 的网络和凭据来源。
   - 不需要：项目 Jenkins；本地 Jenkins、本地 Harbor 已作为平台基础资源维护。
4. 发布/部署详情与远程 Agent 回调收口
   - 交付：`agentTaskId` 关联、任务状态刷新、日志、失败原因、执行记录、发布报告、重试/跳过/人工确认/回滚入口与状态一致性。
   - 用户视角：发布详情和部署详情能完整追踪远程 Agent 执行过程。
   - 环境准备：第 1-3 步完成。
5. 真实发布链路联调
   - 交付：Jenkins Job 发版、Harbor 镜像选择与同步、Agent 执行镜像同步与目标 workload tag 更新。
   - 用户视角：用户可以选择 Jenkins Job 或 Harbor 镜像 tag 发版，不再手工查镜像或改 tag。

## V1 后续研发主线与环境准备（详细说明）

### V1 关键前提调整

- V1 要实现真实项目环境发版/部署，必须先具备可独立部署的项目环境 Agent。
- 研发联调阶段 Agent 部署方式固定为项目环境侧 Linux 主机 + 二进制启动；`docker compose` 只用于后续正式上线部署验证。
- 本地环境按平台侧可连通处理，不需要 Agent；项目环境按平台侧不可连通处理，必须通过 Agent 接入。
- Agent 管理页面生成一次性注册密钥；Agent 首次注册后换长期 Agent token，运行期心跳、任务租约、远程探测和执行回传都必须校验长期 Agent token。
- Agent 支持 `在线 / 待认领`，待认领 Agent 只展示未归属探测摘要，不能进入正式产品/服务视图，也不能执行发布/部署任务。
- Agent 只支持出站访问平台 API，上报心跳、服务列表、镜像版本、运行态快照、任务状态、日志和执行结果，并通过领取/租约机制获取发布/部署任务。
- 在 Agent 不能二进制启动、不能真实注册/心跳/token 校验/探测回传前，不能用 mock 数据替代完成。
- Jenkins、Harbor、Kubernetes 的真实接入必须排在“Agent 可部署 + Agent 出站任务领取链路打通”之后。

### Agent 部署方案

- V1 研发默认方案：Agent 部署在独立 Linux 主机上，使用直接构建出的二进制运行。
- V1 正式上线方案：后续再验证 `docker compose` 运行方式。
- V1 默认不把 Agent 自身作为 Kubernetes 内 workload 部署前提。
- V1 中 Agent 的职责是：
  - 主动访问平台 API 领取/租约获取发布/部署任务和执行数据
  - 主动访问平台 API 上报心跳、服务列表、镜像版本、状态、日志和结果
  - 访问 Jenkins 触发或查询构建任务
  - 访问 Harbor 或兼容 Registry 查询、同步、推送镜像
  - 访问目标 Kubernetes 集群执行运行态采集、部署、更新和健康检查
- 后续是否将 Agent 自身迁移到 Kubernetes，放到 V1 主功能稳定后再评估。

- 联调用例：
  - 至少一个“目标环境缺失服务”的可重复验证场景

建议同时准备的最小联调样本：

- 1 个测试服务对应的 Deployment/StatefulSet manifests
- 如有需要，对应的 Service、ConfigMap、Ingress 等最小配套资源
- 1 套用于验证已有服务更新镜像 tag 的 manifests
- 1 套用于验证目标环境缺失服务首次部署的 manifests

用户视角交付：

- 用户可以从差异页直接识别“目标环境缺失服务”
- 用户可以发起部署并由 Agent 在目标环境完成部署与健康检查
- 用户可以在部署详情页看到每一步结果和失败原因

## 阶段 6：运行态采集与基线闭环（历史拆分，执行顺序以 V1 实现规划为准）

## 阶段 7：差异分类与远程任务闭环（历史拆分，执行顺序以 V1 实现规划为准）

- 详情页展示任务状态、步骤、日志、失败原因

## 阶段 8：项目环境部署与管理达标（历史拆分，执行顺序以 V1 实现规划为准）

目标：达到 V1 最低可交付标准，即可管理项目环境并完成项目环境部署/发布主流程。

外部环境前置要求：

- 进入本阶段前，需要至少有一套可用于联调的项目环境，且提前确认：
  - Agent 部署位置与运行方式

- 项目环境管理和连接状态可见
- Agent 注册、心跳、能力、最近任务可见
- Agent 主动领取发布/部署任务；Agent 上报步骤状态、日志和最终结果
- 项目环境部署任务和发布任务均可从平台发起并追踪
